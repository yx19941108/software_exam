# 案例题模拟考试复盘：DFD、UML补图与下午题保分技巧

## 文档定位

本文件用于沉淀软件设计师案例题模拟考试第一轮、第二轮中暴露出的共性问题，重点不是复述某一道题的标准答案，而是把可迁移的答题方法整理出来，方便后续做真实下午案例题时减少稳定失分。

适用题型：

- DFD / 结构化分析：实体、数据存储、缺失数据流、加工逻辑描述。
- 数据库设计：ER图补联系、关系模式补属性、主外键、m:n联系拆表。
- UML / 面向对象分析设计：参与者、用例、类名、include / extend / generalize关系。
- 算法与代码填空：动态规划变量含义、循环边界、结果计算。
- 设计模式代码填空：接口方法、抽象类、协作调用、关键字拼写。

核心原则：

> 下午题不是自由设计题，优先服从题目给出的图、空位、关系和题干原词。业务解释能通只是候选答案，图形结构能对上才更接近考试答案。

## 一、DFD补数据流题：如何少丢分

### 1. 常见失分原因

本次模拟中，DFD题主要失分不是因为不懂业务，而是因为把“业务上谁和谁有关”直接写成数据流，忽略了DFD图中的加工节点。

典型错误：

- 用户直接把“订餐请求”送到“订单处理”。
- 实际图中应先经过“订餐”加工，再由“订餐”把“订餐请求”交给“订单处理”。
- 把“餐品管理”直接当成用户浏览餐单的加工，但图中用户浏览餐单通常发生在“订餐”加工中，“餐品管理”更偏商户维护餐品信息。

DFD里不能只按业务直觉连线。外部实体、加工、数据存储各有边界：

- 外部实体：系统外的人、组织、外部系统，只能通过数据流与加工交互。
- 加工：真正处理业务动作的节点，数据必须通过加工转换。
- 数据存储：只保存或提供数据，不主动处理业务。

### 2. DFD补流的五步法

#### 第一步：先看父子图平衡

如果题目给了上下文数据流图和0层数据流图，先比较外部实体的输入输出。

上下文图里有：

- 谁给系统什么？
- 系统给谁什么？

0层图里必须能找到对应的数据流入口或出口。如果上下文图有“用户提交订餐请求”，0层图中就必须有某个加工接收用户的订餐请求；如果上下文图有“系统给用户订单状态”，0层图中就必须有某个加工向用户输出订单状态。

父子图平衡是补缺失流的第一优先级。

#### 第二步：每个加工按“四类流”检查

对每个加工问四个问题：

1. 它从外部实体接收什么输入？
2. 它向外部实体输出什么结果？
3. 它需要从哪些数据存储读取数据？
4. 它需要向哪些数据存储写入或更新数据？

例如外卖订餐系统里的“订餐”加工：

- 输入：用户提交订餐请求，选择餐品及数量。
- 读取：餐品信息，用于给用户展示餐单。
- 输出：把订餐请求交给订单处理。
- 写入：生成订单后写入订餐订单信息。

如果图里已有“用户 -> 订餐：订餐请求”，就不要重复写。缺的是“订餐 -> 订单处理：订餐请求”时，答案必须体现“订餐”这个加工。

#### 第三步：用题干动词判断流向

题干动词通常直接提示数据流方向。

| 题干动词 | 常见流向 | 示例 |
| --- | --- | --- |
| 提交、发送、发起、请求 | 外部实体或前一加工 -> 当前加工 | 用户 -> 订餐：订餐请求 |
| 返回、反馈、通知、推送 | 当前加工 -> 外部实体或下一加工 | 订单处理 -> 用户：订单状态 |
| 存储、记录、保存 | 加工 -> 数据存储 | 订单处理 -> 订餐订单信息：更新订单状态 |
| 查询、读取、获取、浏览 | 数据存储 -> 加工，或加工 -> 外部实体 | 餐品信息 -> 订餐：餐单 |
| 更新、修改、删除 | 加工 -> 数据存储 | 配送 -> 订餐订单信息：更新订单状态 |

写答案时尽量用题干原词，不要自己造过于笼统的词。

更稳的写法：

- “更新订单状态”
- “配送码”
- “订餐请求”
- “餐单”

容易扣分的写法：

- “订单状态”
- “订单信息”
- “餐品相关信息”
- “用户请求”

笼统词有时能给部分分，但不如题干原词稳。

#### 第四步：不要让外部实体跳过加工

这是DFD补流最常见的坑。

错误倾向：

```text
用户 -> 订单处理：订餐请求
```

更符合图形结构的写法：

```text
用户 -> 订餐：订餐请求
订餐 -> 订单处理：订餐请求
```

如果图上已经有第一条，缺的通常是第二条。考试问“补缺失数据流”，只写缺的那条。

判断方法：

- 看图里是否已经有一个更靠前的加工承接了外部实体输入。
- 如果有，后续加工的数据一般来自这个加工，不直接来自外部实体。

#### 第五步：数据存储不直接连实体，也不主动干活

数据存储常见连接方式：

```text
加工 -> 数据存储：保存/更新的数据
数据存储 -> 加工：查询/读取的数据
```

高风险写法：

```text
用户 -> 订餐订单信息
订餐订单信息 -> 支付系统
餐品信息 -> 用户
```

如果题图确实要求用户看到餐单，一般也应写为：

```text
餐品信息 -> 订餐：餐单
订餐 -> 用户：餐单
```

数据存储本身不主动给用户展示东西，必须经由加工。

### 3. DFD补流答题模板

真实考试中，类似“补充缺失的数据流及其起点和终点”的题，通常不要求完整重画图。更稳的答案是三列表格：

| 数据流名称 | 起点 | 终点 |
| --- | --- | --- |
| 订餐请求 | 订餐 | 订单处理 |
| 更新订单状态 | 订单处理 | 订餐订单信息 |
| 配送码 | 配送 | 用户 |
| 餐单 | 餐品信息 | 订餐 |

如果担心老师看不懂代号，建议写“代号 + 名称”：

```text
订餐请求：P3订餐 -> P4订单处理
更新订单状态：P4订单处理 -> D2订餐订单信息
配送码：P5配送 -> E3用户
餐单：D3餐品信息 -> P3订餐
```

这样既对齐图，又有真实名称。

### 4. DFD加工逻辑描述技巧

加工逻辑描述题常要求“采用结构化语言”。结构化语言不是写作文，而是把题干转成顺序、选择、循环。

常用结构：

```text
收到订餐请求后：
    向外卖平台请求配送；
    外卖平台发布配送单，由骑手接单；
    外卖平台返回接单状态；
    IF 接单成功 THEN
        向支付系统发送支付请求；
        接收支付状态；
        IF 支付成功 THEN
            更新订单状态为已接单；
            向商户发送订餐请求；
            向用户发送订单状态；
        ELSE
            更新订单状态为下单失败；
            向外卖平台请求取消配送；
            向用户发送下单失败；
        ENDIF
    ELSE
        更新订单状态为下单失败；
        向用户发送下单失败；
    ENDIF
```

保分要点：

- 不要漏中间判断依据，如“外卖平台发布配送单、骑手接单、返回接单状态”。
- IF和ELSE要成对。
- 支付失败和接单失败是两个不同失败分支。
- 用题干原词，不要换成太口语的“失败了就通知一下”。

## 二、数据库设计题：ER图和关系模式保分

### 1. ER图补联系的核心

数据库案例题常见问法：

- 补充实体之间的联系。
- 标注联系类型：1:1、1:n、m:n。
- 补关系模式中的缺失属性。
- 给出主键和外键。

判断联系类型时，题干里的数量词最重要。

| 题干表述 | 关系判断 |
| --- | --- |
| 一个客户至少有一辆车，一辆车只属于一个客户 | 客户 1:n 车辆 |
| 每个部门有多名员工，每名员工只属于一个部门 | 部门 1:n 员工 |
| 一个维修工可接多张维修单，一张维修单只对应一个维修工 | 维修工 1:n 维修单 |
| 一个学生可选多门课，一门课可被多个学生选 | 学生 m:n 课程 |

### 2. 1:n关系转关系模式

规则：

> 将1端主键加入n端关系，作为n端外键。

例：

```text
客户 1:n 车辆
客户(客户号, ...)
车辆(车牌号, 客户号, 车型, 颜色, 车辆类别)
```

车辆表中的“客户号”不是车辆本身的自然属性，而是从1端客户关系迁移过来的外键。

### 3. m:n关系必须拆成中间关系

规则：

> m:n关系不能简单把一方主键塞到另一方表里，必须新增中间关系。

例：

原来：

```text
维修工 1:n 维修单
```

如果变成“一张维修单需要多个维修工处理”，则：

```text
维修工 m:n 维修单
```

应新增：

```text
维修(员工号, 维修单号, 维修地点, 维修时间)
```

其中：

- 主键通常是 `(员工号, 维修单号)`，如果题干给出日期、序号等，也可能一起组成主键。
- `员工号` 是外键，引用员工。
- `维修单号` 是外键，引用维修单。

### 4. 主键、外键答题模板

```text
车辆关系：
主键：车牌号
外键：客户号，引用客户(客户号)

维修单关系：
主键：维修单号
外键：车牌号，引用车辆(车牌号)；
     维修员工号，引用员工(员工号)
```

注意：

- 主键是“唯一标识本关系元组”的字段。
- 外键是“从别的关系迁移过来，用来表示联系”的字段。
- 如果题干说“某某号唯一标识某关系”，通常就是主键。

## 三、UML填图题：不要只按业务解释，要先服从图

### 1. 本次暴露的问题

UML题最容易出现“业务上说得通，但图上不对”的情况。

例如线上抓药APP题中，题干里确实有两组可泛化的业务：

```text
取药方式 -> 自行到店取药 / 送药上门
支付 -> 微信支付 / 支付宝支付
```

如果只按业务描述，两组都能解释通。但考试不是自由建模，而是让考生根据给定用例图和类图填空。

图里的空位如果形成“一父二子”的泛化结构，就要优先找题干中最能匹配该结构、且与图中其他关系相连的那一组。

该题标准锁定为：

```text
支付 -> 微信支付 / 支付宝支付
支付方式 -> 微信支付 / 支付宝支付
```

所以“取药方式”不是业务错，而是不符合这张图的空位意图。

### 2. UML填图四步法

#### 第一步：先识别空位类型

先看题目问什么：

- A：参与者，通常是系统外的人或外部系统。
- U：用例，通常是用户想通过系统完成的目标。
- C：类，通常是系统内部对象、业务概念或抽象类型。

不要把属性、动作、业务方式混进类名或参与者名。

#### 第二步：先看图形关系，再读题干

图形关系比业务感觉更重要。

| 图形关系 | 应优先找什么 |
| --- | --- |
| 泛化箭头 | 父类/子类，父用例/子用例 |
| include | 必做的公共子用例 |
| extend | 条件触发的可选扩展 |
| 普通关联 | 题干中直接交互的对象 |
| 聚合/组合 | 整体-部分关系 |

如果图里是泛化结构，就优先找“某某是一种某某”。

例：

```text
微信支付 是一种 支付
支付宝支付 是一种 支付
```

这比“确认处方后要支付”更符合泛化结构。

#### 第三步：按候选组匹配空位数量

如果图上是：

```text
C1
├── C2
└── C3
```

就去题干里找“一父二子”的候选组。

候选组可能有多个：

```text
支付方式 -> 微信支付 / 支付宝支付
取药方式 -> 到店取药 / 送药上门
```

此时继续看：

- 哪一组在用例图中也出现？
- 哪一组和其他类有关联？
- 哪一组更可能是系统内部类？
- 题目标准术语更倾向哪一组？

如果类图和用例图都出现支付泛化，那么选支付组更稳。

#### 第四步：用题干原词，不要创造新词

题干说“支付方式、微信、支付宝”，答案就写：

```text
支付方式
微信支付
支付宝支付
```

不要改成：

```text
费用计算方式
线上支付
第三方支付
```

这些可能业务上合理，但和标准答案不稳定。

### 3. include、extend、generalize区别

#### include：必做的公共步骤

含义：

> 一个用例执行时，必然包含另一个用例。

例：

```text
确认处方 include 支付
```

如果确认处方流程中必然要进入支付动作，就可以用include。include强调“必然执行”和“公共行为抽取”。

#### extend：条件触发的可选扩展

含义：

> 基本用例本身可以独立完成；满足某个条件时，才执行扩展用例。

例：

```text
确认处方 extend 填写收货地址
```

如果只有选择送药上门时才填写收货地址，那么填写收货地址就是条件扩展。

extend不是继承，也不是“必须发生在之后”。

#### generalize：泛化/继承

含义：

> 子用例或子类是父用例或父类的特殊形式。

例：

```text
微信支付 是 支付 的一种
支付宝支付 是 支付 的一种
```

看到“多种具体形式”“不同方式”“不同类型”，要考虑泛化。

### 4. UML保分答题模板

如果不确定，可以在答案旁写简短理由：

```text
U2：支付
U3：微信支付
U4：支付宝支付
理由：U3、U4是U2的具体支付方式，图中为泛化关系。
```

这样即使名称略有偏差，阅卷时也更容易给部分分。

## 四、算法与代码填空：动态规划题保分

### 1. 动态规划题的识别信号

题干出现以下词时，高概率是动态规划：

- 最优子结构
- 子问题
- 自底向上
- 递归式
- 保存中间结果
- cost、dp、trace、path等数组

矩阵链乘问题就是典型动态规划。

### 2. 矩阵链乘变量含义

常见变量：

```text
n：矩阵个数
seq[]：维度序列
cost[i][j]：从第 i+1 个矩阵乘到第 j+1 个矩阵的最小代价
trace[i][j]：最优断开位置
p：当前子链长度差
i：子链起点
j：子链终点
k：断开位置
```

循环关系：

```text
j = i + p
k 从 i 枚举到 j-1
```

因此常见填空：

```c
j = i + p;
k < j;
cost[i][k] + cost[k + 1][j] + seq[i] * seq[k + 1] * seq[j + 1];
trace[i][j] = tempTrace;
```

### 3. 防止混淆的口诀

```text
p 是长度，不是下标边界；
j 是右端点；
k 是断点，只能断到 j 前面。
```

如果写成 `k < p`，就把“长度变量”和“下标变量”混了。

### 4. 复杂度判断

代码有三层循环：

```text
p循环
  i循环
    k循环
```

所以时间复杂度是：

```text
O(n^3)
```

## 五、设计模式代码填空：Java题保分

### 1. 中介者模式的核心

中介者模式用于把多个对象之间的复杂交互，集中交给一个中介者对象处理。

题干中的支付平台整合：

- Amazon和Ebay不直接互相调用。
- 它们持有同一个 `WebServiceMediator`。
- 具体购买时调用 `mediator.buy(money, this)`。

核心结构：

```java
interface WebServiceMediator {
    void buy(double money, WebService service);
}

abstract class WebService {
    protected WebServiceMediator mediator;
    abstract void buyService(double money);
}
```

具体同事类：

```java
class Amazon extends WebService {
    public void buyService(double money) {
        mediator.buy(money, this);
    }
}
```

### 2. 代码填空最容易丢的分

#### 关键字拼写

Java关键字必须拼对：

```java
abstract
```

写成：

```java
abstarct
```

即使思路对，也会导致代码无法编译，考试通常不给这一空。

#### 方法签名要完整

接口方法、抽象方法要包含：

- 返回类型
- 方法名
- 参数列表

例如：

```java
void buy(double money, WebService service)
abstract void buyService(double money)
```

只写方法名或漏参数，容易扣分。

#### this的含义

```java
mediator.buy(money, this)
```

这里的 `this` 表示当前具体平台对象，如Amazon或Ebay。中介者根据传入的具体对象判断应该调用哪个平台逻辑。

## 六、后续练习建议

### 1. DFD专项训练方式

每做一道DFD题，强制输出下面四项：

```text
外部实体清单：
数据存储清单：
每个加工的输入/输出/读库/写库：
缺失数据流三列表：
```

不要直接写答案，先做这四步。练5题后，补流稳定性会明显提升。

### 2. UML专项训练方式

每做一道UML题，强制输出：

```text
空位类型：参与者 / 用例 / 类
空位所在关系：泛化 / include / extend / 关联 / 聚合组合
题干候选词分组：
最终选择及图形理由：
```

尤其要训练“图形理由”，不能只写业务理由。

### 3. 做题时的自检问题

DFD自检：

- 我写的数据流是否通过加工节点？
- 起点和终点是否是图里已有元素？
- 数据存储有没有直接连外部实体？
- 数据流名称是否用了题干原词？
- 是否先检查了父子图平衡？

UML自检：

- 这个空问的是参与者、用例还是类？
- 这个空周围是什么关系？
- 是否存在“一父二子”的泛化候选组？
- 我的答案是业务上能通，还是图上也能通？
- 是否用了题干原词？

### 4. 当前最需要强化的能力

当前不是知识点完全不会，而是需要把答题习惯从“业务解释优先”调整为：

```text
先服从图形结构
再匹配题干原词
最后解释业务逻辑
```

这是DFD和UML少丢分的核心。
