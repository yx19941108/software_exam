# 第16课：设计模式专项

## 本课定位

设计模式专项属于下午案例分析阶段。本课不按某一年真题编号推进，而是训练“看到需求变化、类图结构或代码空缺后，判断模式、补全角色、说明理由”的能力。

本课只讲设计模式案例题的答题方法，不放 paired 训练题的答案、解析或结论提示。配套训练题单独放在 `doc/Software-Designer-master/真题/xisai_md/真题训练/`。

本课本地题源依据来自 `doc/Software-Designer-master/真题/xisai_md/xisai_md_总索引.md` 和 `doc/Software-Designer-master/真题/xisai_md/题目/` 中的案例题检索结果。近年完整案例优先；`2024下半年案例题`、`2025上半年案例题` 这类回忆版或答案不完整材料只作为补充，不作为第一模板基线。

## 本课目标

完成本课后，需要达到以下能力：

1. 能把设计模式题分成三类：模式识别题、代码填空题、类图角色题。
2. 能从题干中的“变化点”判断应该把什么封装起来，例如状态变化通知、算法切换、整体-部分层次、接口不兼容、动态增加职责、请求逐级处理。
3. 能把常见模式的关键角色对应到类图或代码中，例如抽象主题/具体主题、抽象策略/具体策略、抽象构件/叶子/容器、目标接口/适配器/被适配者。
4. 能写出 50 到 150 字的选择理由，做到“模式名 + 需求特征 + 模式意图 + 适配原因”四件事齐全。
5. 能在代码填空题中先找接口方法、继承/组合关系、成员变量和调用点，再填具体语句，避免只背模式名。

## 前置知识

设计模式是对常见设计问题的复用解法。考试不会要求完整背 23 种模式的定义，但会反复考“需求特征和模式意图是否匹配”。

常见分类：

- 创建型模式：关注对象怎样创建，例如工厂方法、抽象工厂、单例、原型、建造者。
- 结构型模式：关注类或对象怎样组合，例如适配器、桥接、组合、装饰、外观、享元、代理。
- 行为型模式：关注对象之间怎样协作和分配职责，例如策略、观察者、模板方法、命令、状态、责任链、访问者、中介者、迭代器、备忘录、解释器。

本地案例题库统计口径：统计 `doc/Software-Designer-master/真题/xisai_md/题目/*案例题.md` 中的 2015-2025 案例题条目，共 103 道。第16课当前按 Java 路线训练，因此只统计题干明确写有 `Java代码` / `Java 代码` 的 GoF 设计模式案例题，共 16 道，占全部案例题 15.5%。同一业务场景下的 Java / C++ 成对题视为选做版本，只保留 Java 版本计数，C++ 版本不计入；“算法设计策略”“状态图/状态字段”“组合主键”等非 GoF 设计模式误命中不计入；只问“新增哪种设计模式”的非 Java 题暂不计入 Java 训练权重。

从 Java 训练路线看，优先级最高的是策略，其次是建造者/生成器；观察者、组合、访问者、适配器、装饰、桥接、状态、享元、中介者、外观、备忘录在 Java 题库中各有 1 道，适合作为轮换覆盖。责任链当前本地题库中有非 Java 案例问法，但不计入 Java 专项统计，可作为概念补充而不是 Java 代码训练主线。

下午案例题更常见的不是纯定义，而是把模式放进业务场景或代码中考：

- 需求新增：系统要支持某种变化，问增加哪种设计模式并说明原因。
- 类图识别：给出类图和若干类名，问类、属性、方法或模式角色。
- 代码填空：给出 C++ 或 Java 代码，空缺通常落在抽象接口、具体实现、组合引用、构造绑定、方法调用处。

判断设计模式时先抓“变化点”：

- 多个对象监听同一个对象变化：关注状态变化后的自动通知。
- 多种算法、规则、计费、换算、行为可互换：关注算法族封装。
- 单个对象和组合对象要被一致处理：关注树形整体-部分结构。
- 旧接口和新接口不一致但又要复用已有类：关注接口转换。
- 给对象动态叠加功能，且不想大量增加子类：关注职责叠加。
- 请求要沿着多个处理者逐级传递：关注处理链。
- 需要在不改元素类的前提下增加操作：关注访问操作外置。

## 核心考法识别

模式识别题的核心不是“背出模式名”，而是把题干特征翻译成模式意图。答题时不要只写“因为符合某模式”，要写清楚“哪里符合”。

常见识别信号：

- 发生变化后通知多个依赖对象：通常考“一对多依赖”和“自动更新”。
- 不同促销、不同换算、不同刹车痕迹、不同规则可以替换：通常考“算法族封装”和“运行时选择”。
- 菜单、组织、团体、目录这类可递归包含的结构：通常考“整体-部分”和“一致使用”。
- 支付接口、第三方接口、旧系统接口需要统一访问：通常考“接口适配”或“中间协调”，要看重点是转换接口还是协调多个对象交互。
- 发票抬头、正文、脚注这类功能层层叠加：通常考“动态增加职责”。
- 应急处理、审批、过滤、逐级转交：通常考“多个处理者依次处理请求”。
- 图书、论文、菜单项等稳定元素上新增统计、打印、导出等操作：通常考“元素结构稳定，操作经常新增”。

代码填空题的核心是角色落点：

- 先看类图中的抽象类或接口，它通常对应模式中的抽象角色。
- 再看具体子类，它们通常对应具体角色。
- 找成员变量：如果一个类持有抽象接口类型的成员变量，通常说明它通过组合调用可替换对象。
- 找构造函数：构造函数经常负责绑定具体策略、具体观察者、装饰对象或组合对象。
- 找循环或集合：集合常用于保存观察者、子构件或处理者列表。
- 找统一调用方法：如 `update`、`accept`、`execute`、`handle`、`operation`、`print`、`stop` 等，通常是填空高频点。

类图角色题的核心是“不要只看类名，要看关系”：

- 泛化/实现关系表示抽象角色和具体角色。
- 聚合/组合关系表示对象持有另一个对象，常见于策略、装饰、组合、观察者。
- 依赖关系表示临时使用，常见于访问者、命令、工厂方法。
- 多重度可以辅助判断“一对多”“树形结构”“集合保存”等。

## 固定答题步骤

第一步，先判题型。

如果题目问“新增需求适合哪种模式并说明原因”，先按模式识别题处理。如果题目给出代码空缺，先按代码填空题处理。如果题目给出类图并要求 C1、C2 或方法属性，先按类图角色题处理。

第二步，圈出变化点。

不要一上来套模式名。先用一句话写出题干中真正变化的东西：是算法会变、对象状态会变、接口不一致、功能要叠加、结构可递归、请求要逐级处理，还是操作经常新增。

第三步，匹配模式意图。

把变化点和模式意图对应起来。判断标准是“这个模式解决的核心问题，是否正好是题干的核心变化”。如果只是名字相似，不算匹配。

第四步，落到类图或代码。

如果有类图，找抽象角色、具体角色、客户类、被持有对象、集合关系。如果有代码，找接口方法、具体实现、构造函数、成员变量和调用点。

第五步，组织答案。

模式说明题建议按这个模板写：

`采用 [模式名]。题干中的 [需求特征] 表明 [变化点]，该模式的意图是 [模式意图]，可以通过 [角色或结构] 使 [变化点] 独立变化 / 自动通知 / 统一处理 / 动态扩展，因此适合。`

代码填空题建议按这个顺序检查：

1. 抽象接口缺方法时，先找所有具体类已经实现了什么方法。
2. 成员变量缺类型时，优先填抽象类型，而不是具体类。
3. 构造函数缺语句时，检查是否要绑定具体对象、添加观察者、包装被装饰对象或加入子构件。
4. 调用语句缺失时，优先调用抽象接口中的统一方法。
5. 如果 C++ 题出现指针，注意 `->`；Java 题注意对象方法调用和集合类型。

## 判分点模板

模式识别题通常按 3 分计，建议拆成：

- 1 分：模式名正确。
- 1 分：说出题干中的关键需求特征。
- 1 分：说明该模式如何解决这个特征。

不要只写“采用某模式，因为它适合该场景”。这类答案通常只能拿模式名分，理由分不稳。

代码填空题通常按 5 分计，常见空位包括：

- 抽象方法声明。
- 具体子类方法实现。
- 抽象类型成员变量。
- 构造函数中的对象绑定。
- 通过抽象接口调用统一方法。

类图角色题通常按类名、属性、方法逐项给分：

- 类名题要使用题干或图中给定词汇，不要自由翻译。
- 属性题只写关键属性，避免多写无关属性。
- 方法题要看已有代码或类图标注，不要凭业务感觉补方法。
- 多空题要保持编号清晰，便于阅卷。

理由表达模板：

- “一对多通知”类理由：强调一个对象状态变化后，多个依赖对象需要同步更新。
- “算法替换”类理由：强调多个规则或算法可以互换，调用方不应依赖具体实现。
- “整体-部分”类理由：强调单个对象和组合对象需要一致处理。
- “接口转换”类理由：强调已有类接口与目标接口不一致，需要转换后复用。
- “动态职责”类理由：强调不改变对象结构的情况下动态增加功能。
- “逐级处理”类理由：强调请求沿处理链传递，直到被合适对象处理。

## 常见失分点

1. 只写模式名，不写题干特征和模式意图。
2. 把策略模式和状态模式混淆：策略关注“算法/规则可替换”，状态关注“对象内部状态改变导致行为改变”。
3. 把适配器模式和中介者模式混淆：适配器重点是“接口转换”，中介者重点是“对象之间交互集中协调”。
4. 把组合模式和装饰模式混淆：组合重点是“树形整体-部分”，装饰重点是“动态增加职责”。
5. 把观察者模式写成“通知用户”就结束，没有说明“一对多依赖”和“状态变化自动通知”。
6. 代码填空时填具体类，导致扩展性变差；模式题通常更希望通过抽象类型解耦。
7. C++ 和 Java 语法混写，例如把 `->`、`.`、抽象方法声明、接口方法声明写错。
8. 类图题多写、错写、自由改名，导致本来会的空被扣分。
9. 看到题干中出现某个模式名就机械照抄，不回到代码上下文验证空缺位置。
10. 理由超过字数限制，写了很多泛泛定义，却没有命中题目需求。

## 过关标准

本课过关不看是否能背完整 23 种设计模式，而看是否能稳定拿到下午题中的可得分：

1. 模式识别题：能在 3 分题中稳定拿到模式名分和理由分，目标 2.5 分以上。
2. 代码填空题：能先从类图和上下文找角色，再补接口、成员变量、构造绑定和调用语句，目标 5 分题拿到 3.5 分以上。
3. 类图角色题：能按题干给定词汇作答，不自由扩写，目标 8 分题拿到 6 分以上。
4. 文字理由：能在 50 到 150 字内写清“需求特征 + 模式意图 + 适配原因”。
5. 三轮训练后，如果仍反复混淆“策略/状态”“适配器/中介者”“组合/装饰”“观察者/责任链”，需要追加专项辨析题，不直接进入第17课。
