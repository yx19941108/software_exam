# 第13课：DFD / 结构化分析专项

## 本课定位

本课是下午案例题专项线的第 1 课，目标不是泛泛复习“软件工程”或“系统分析”，而是专门攻克数据流图类大题的稳定得分点。  
这一类题的核心特点是：题面业务背景会变，但判分动作高度稳定，通常围绕 `实体`、`数据存储`、`缺失数据流`、`数据组成`、`加工逻辑` 这几类小问反复出现。  
因此，第13课要解决的不是“看懂一个具体行业案例”，而是把你看到任意 DFD 题时的第一反应训练成固定动作。

## 本课目标

学完本课后，你要达到下面 4 个结果：

1. 看到题目出现 `上下文图 / 顶层图 / 0层图 / 数据流图` 这些标志词时，能立刻确认这是 DFD 题，而不是数据库题或 UML 题。
2. 能按照固定顺序处理常见四类设问：
   - 补实体
   - 补数据存储
   - 补缺失数据流
   - 补数据组成或加工逻辑
3. 能把“父图-子图平衡”“外部实体只和加工连”“数据流名称像数据、不像动作”这些判分规则写进答案，而不是只靠感觉猜。
4. 能在限时状态下先搭答题骨架，再填具体词语，避免因为业务背景陌生而乱写。

## 前置知识

进入本课前，你不需要先掌握复杂建模理论，但至少要接受下面 3 个前提：

1. 数据流图不是程序流程图，它描述的是“数据在系统中的流动和处理”，不是代码执行步骤。
2. DFD 题的业务背景可以很陌生，但绝大多数答案都能直接从题干原词中抽取，不需要凭空发明术语。
3. 下午题判分重在“结构对、术语对、方向对”，不是写得越多越好。

如果这 3 条还不稳，本课就先把术语和题型识别建立起来，不追求一上来就全对。

## 核心考法识别

DFD / 结构化分析题最常见的考法有 5 类：

### 1. 实体补全

典型问法：

- 给出图中的 `E1~E4` 名称
- 使用说明中的词语给出外部实体名称

识别信号：

- 实体一般是系统外部的人、设备、第三方系统、组织角色
- 实体不会是系统内部加工步骤

### 2. 数据存储补全

典型问法：

- 给出 `D1~D4` 的名称
- 根据说明补充数据存储

识别信号：

- 数据存储常以 `文件 / 表 / 记录 / 信息` 结尾
- 命名通常来自题干里的稳定名词，而不是动作短语

### 3. 缺失数据流补全

典型问法：

- 补充缺失数据流及其起点终点
- 根据说明和图中术语补全缺失连线

识别信号：

- 这是 DFD 题里最容易失分、也是最关键的部分
- 先看父图-子图平衡，再看题干中明确说“谁把什么给谁”

### 4. 数据组成补全

典型问法：

- 给出某个数据流的组成
- 写成 `{字段1 + 字段2 + 字段3}` 形式

识别信号：

- 答案几乎都来自题干列举的数据项
- 不能漏掉标识类字段和时间类字段

### 5. 加工逻辑描述

典型问法：

- 用结构化语言描述某加工逻辑
- 说明某加工可分解为哪些子加工
- 进一步分解时应注意什么错误

识别信号：

- 常见输出是 `IF / THEN / ELSE / ENDIF`、`DO / WHILE / ENDDO`
- 若题目问分解错误，高频就是 `黑洞 / 奇迹 / 灰洞`

## 固定答题步骤

做 DFD 题时，固定按下面顺序处理，不要一上来就逐空乱填。

### 第一步：先给题目定型

先确认这是一道 `结构化分析 / 数据流图` 题。只要看到：

- 上下文图
- 顶层图
- 0层图
- 外部实体
- 数据存储

就进入 DFD 模板，不要把思路带去数据库或 UML。

### 第二步：先通读题干功能列表

把题干里的每一项功能按下面三类做心里标记：

- 谁是系统外部角色或外部设备
- 系统内部做了哪些加工
- 会沉淀哪些数据

这一步不急着答题，先做“名词池”。

### 第三步：优先解实体题

先做实体，是因为实体最稳定、最容易从题干直接定位。  
判断规则：

1. 看是不是系统外的人、设备、组织、第三方服务。
2. 看它是不是直接和系统交换数据。
3. 不要把系统内部加工误写成实体。

### 第四步：再解数据存储题

判断规则：

1. 先找题干里已经出现的稳定名词。
2. 再看图中哪个加工在“写入”和“读取”同一类信息。
3. 命名时优先使用题干原词，再补 `文件 / 表 / 信息 / 记录` 这类存储后缀。

### 第五步：再做缺失数据流

这是主战场，必须固定两轮检查：

1. 平衡检查：
   - 父图有、子图没有
   - 子图关键输入输出无法落回父图
2. 题干动作检查：
   - 谁发起
   - 发的是什么数据
   - 发给谁

只要题干明确写了“根据 X 对 Y 处理”“向 Z 发送 W”“从 A 获取 B”，就优先考虑是否有遗漏数据流。

### 第六步：遇到数据组成题，直接回题干找字段

写法固定成集合式：

`数据流 = {字段1 + 字段2 + 字段3}`

先收标识字段，再收业务字段，再检查是否漏掉时间或状态字段。

### 第七步：遇到加工逻辑题，写结构化语言骨架

不要直接写散文。  
先写控制结构，再填业务句子，例如：

```text
IF (条件)
THEN
    处理1
ELSE
    处理2
ENDIF
```

如果题目问子加工分解，就写“先做什么，再做什么”；  
如果题目问分解错误，就先回忆 `黑洞 / 奇迹 / 灰洞`。

## 判分点模板

下面这套模板是第13课之后每次做 DFD 题都要先套上的。

### 模板 A：实体题

答题时先自问：

1. 这是系统外部对象吗？
2. 它和系统之间有真实数据交换吗？
3. 题干里有没有直接对应的角色词或设备词？

只要三问里前两问都成立，答案大概率就在题干原词里。

### 模板 B：数据存储题

答题时先自问：

1. 这个数据是被长期保存还是瞬时传递？
2. 哪个加工写入它，哪个加工读取它？
3. 命名应该落在“规则 / 信息 / 记录 / 文件 / 表”哪一类？

### 模板 C：缺失数据流题

答题时按固定句式思考：

`根据题干，起点需要把什么数据发送给终点，因此应补充数据流 X，起点是 A，终点是 B。`

这样写时，你脑子里会自动补齐：

- 数据流名称
- 起点
- 终点

### 模板 D：数据组成题

先从题干中抄出所有已经明确列举的数据项，再删去无关描述词，只保留字段名。  
优先级通常是：

1. 编号/ID
2. 类型/状态
3. 业务数据
4. 时间

### 模板 E：加工逻辑题

如果问“描述逻辑”，你的答案至少要有：

1. 触发条件
2. 分支判断
3. 处理结果
4. 输出动作

如果问“分解错误”，直接回到：

1. 黑洞：只有输入没有合理输出
2. 奇迹：没有输入却冒出输出
3. 灰洞：输入不足以推出给定输出

### 模板 F：分层 DFD 的三类分解错误

当题目问“进一步分解时应注意哪三种常见错误”时，考的不是业务名词，而是分层数据流图的合法性。核心原则是：子图必须能解释父图中的加工，不能让数据凭空消失，也不能让数据凭空出现。

#### 1. 黑洞

黑洞指一个加工有输入，但没有与输入相匹配的输出。  
直观理解：数据进了这个加工以后就“没了”，系统没有交代处理结果去哪里。

判断方法：

1. 看某个加工是否接收了数据流。
2. 再看它是否产生了合理输出、写入数据存储，或把处理结果传给其他加工/外部实体。
3. 如果只有输入没有输出，就优先判断为黑洞。

常见失分场景：题干明明说某功能会“生成结果、保存结果、展示结果、发送通知”，但 0 层图或子图里只画了输入，没有画出结果数据流。

#### 2. 奇迹

奇迹指一个加工没有输入，却产生了输出。  
直观理解：系统没有任何数据来源，却突然给出了结果，就像“凭空变出来”。

判断方法：

1. 看某个加工是否向外输出了数据流。
2. 再反查它有没有足够的输入数据、读取数据存储，或接收外部实体提供的信息。
3. 如果完全没有输入来源，却能输出业务结果，就优先判断为奇迹。

常见失分场景：图上某加工直接输出报表、预测、统计结果、判定结果，但没有任何基础数据、规则或查询条件进入该加工。

#### 3. 灰洞

灰洞指一个加工虽然有输入，也有输出，但输入不足以推出输出。  
它和黑洞、奇迹的区别是：灰洞看起来“有进有出”，但仔细检查会发现输入数据不够，不能支撑输出结果。

判断方法：

1. 先列出输出结果需要哪些基础数据或规则。
2. 再检查这些数据或规则是否真的流入了该加工。
3. 如果输入只够完成一部分处理，却输出了更复杂的结果，就优先判断为灰洞。

常见失分场景：加工要输出统计、预测、分析、判定这类结果，但图里只给了原始数据，没有给规则；或者只给了实时数据，没有给历史数据；或者只有编号，却输出完整详情。

#### 答题句式

如果题目只问名称，直接写：

`黑洞、奇迹、灰洞`

如果题目要求说明含义，可以写：

`黑洞是只有输入没有合理输出；奇迹是没有输入却产生输出；灰洞是虽有输入和输出，但输入不足以推出该输出。`

## 常见失分点

第13课最容易出现的失分，不是完全不会，而是下面这些细小但致命的错误：

1. 把系统内部加工名称错当成外部实体。
2. 数据流名称写成“管理”“处理”“控制”这类动作词，而不是数据名词。
3. 只看 0 层图，不做父图-子图平衡检查，导致漏补关键数据流。
4. 数据组成题漏掉 `编号`、`时间` 这类低调但高频得分项。
5. 加工逻辑题直接写自然语言大段解释，没有结构化控制骨架。
6. 看到陌生行业背景就慌，开始自造术语，脱离题干原词。
7. 分不清“存储的数据”和“传递的数据”，把两者写混。

## 过关标准

第13课学完后，至少要达到下面 5 条，才算这课真正过关：

1. 看到 DFD 题，能在 30 秒内说出本题大概率会考哪几类小问。
2. 能不看答案，先写出“实体 -> 存储 -> 缺失流 -> 数据组成/加工逻辑”的答题顺序。
3. 做缺失数据流题时，能先做一次父图-子图平衡检查，而不是直接猜。
4. 遇到加工逻辑题，能主动用 `IF / THEN / ELSE` 或 `DO / WHILE` 组织答案。
5. 做完一题后，能明确指出自己错在：
   - 题型没识别
   - 术语没对齐
   - 数据流方向写反
   - 漏了字段
   - 逻辑骨架没搭

如果这 5 条还做不到，就说明第13课还没真正学透，后面不能急着进入下一类下午大题。
