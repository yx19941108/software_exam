# 第2章 程序设计语言基础知识

## 适用范围

- 本文件只沉淀当前会话中已经通过真题练习、判分和讲解覆盖到的知识点。
- 本文件不是对整章教材的完整重写，不把本轮未覆盖内容伪装成“已经学完”。
- 本轮额外补充了为应对真题而必须掌握的 Python 类型与运算基础，但范围仍以考试常考点为限。

## 本轮已覆盖知识点

### 1. 程序设计语言与语言处理程序

- 程序语言题在软件设计师上午题里会混合考：
  - Python 基础语法与容器
  - Java 语言特征
  - 编译程序、解释程序、汇编程序
  - 文法、后缀式、参数传递
- 汇编程序的作用是把汇编语言源程序翻译成机器语言程序或目标代码。
- 编译程序的典型特征：
  - 先翻译后执行
  - 执行效率通常高
  - 生成的目标程序依赖具体平台
- 解释程序的典型特征：
  - 边解释边执行，或先转成字节码再由解释器/虚拟机执行
  - 通常不生成可独立运行的目标程序
  - 执行效率通常低于直接运行编译后的机器代码

### 2. 编译程序工作过程

- 编译程序的核心处理链路要记成固定顺序：
  - 源程序字符流
  - 词法分析
  - 记号流
  - 语法分析
  - 语义分析
  - 中间代码生成
  - 代码优化
  - 目标代码生成
- 词法分析：
  - 输入是字符流
  - 输出是记号流
  - 负责把源码拆成关键字、标识符、常量、运算符、分隔符等 token
- 语法分析：
  - 输入是记号流
  - 负责检查语句结构是否符合文法
  - 括号不匹配、表达式结构错误通常在这一阶段发现
- 语义分析：
  - 负责检查“意思是否正确”
  - 类型检查属于语义分析
  - 变量是否先定义后使用、参数类型是否匹配也属于这一类检查

### 3. 表达式与后缀式

- 中缀表达式：
  - 运算符写在两个操作数中间
  - 更接近人平时写算式的习惯
  - 例如 `a+b`、`10*(40-30/5)+20`
- 后缀表达式：
  - 运算符写在操作数后面
  - 不依赖括号也能确定运算顺序
  - 例如 `a b +`、`10 40 30 5 / - * 20 +`
- 两者最核心的区别：
  - 中缀式需要结合括号和优先级判断先算谁
  - 后缀式按从左到右扫描并结合栈处理即可
- 真题里考后缀式，本质上是在考：
  - 你是否能识别运算顺序
  - 你是否会用栈模拟表达式求值
- 中缀表达式转后缀式时，要按：
  - 括号
  - 运算符优先级
  - 结合顺序
  依次处理。
- 典型题型包括：
  - 直接写出后缀式
  - 根据后缀式求值
  - 根据后缀式求操作数栈的最小容量
- 对表达式 `10*(40-30/5)+20`：
  - 后缀式是 `10 40 30 5 / - * 20 +`
  - 用栈求值时，操作数栈最大深度为 `4`

### 3.1 中缀表达式与后缀表达式对比

- 同一个表达式的两种写法：
  - 中缀式：`a+b*c`
  - 后缀式：`a b c * +`
- 为什么后缀式不需要括号：
  - 因为运算符总在“本次参与运算的操作数”后面出现
  - 扫描到运算符时，栈顶最近的两个操作数就是它的运算对象
- 做题时的最稳方法：
  - 看到中缀式，先按优先级和括号找“谁先算”
  - 谁先算，就先在后缀式里把对应操作数和运算符凑成一组
- 快速对比例子：
  - 中缀式：`a-b*c`
  - 因为先算 `b*c`
  - 后缀式：`a b c * -`
- 再看一个带括号例子：
  - 中缀式：`(a-b)*(c+d)`
  - 先算 `a-b`，再算 `c+d`，最后做乘法
  - 后缀式：`a b - c d + *`
- 若题目反过来给后缀式求值：
  - 遇到操作数就入栈
  - 遇到运算符就弹出两个操作数计算
  - 结果再压回栈中
- 混淆根源通常有两个：
  - 把“书写形式”与“真实运算顺序”混在一起
  - 看见后缀式时，没有切换到“栈处理”的思路

### 4. 参数传递

- 传值调用：
  - 形参得到的是实参值的副本
  - 形参变化通常不会反映到实参
  - 实参可以是变量、常量、表达式
- 引用调用：
  - 形参与实参绑定到同一对象或同一存储单元语义上
  - 形参变化可以反映到实参
  - 真题里经常用来考“函数返回后原变量有没有被改”
- 做这类题时，先盯清楚：
  - 哪个参数是传值
  - 哪个参数是引用
  - 函数内部改的是形参副本，还是实参本体

### 5. Java 语言特征

- Java 的典型运行特征包括：
  - 源程序先编译成字节码
  - 由 JVM 执行
  - 采用解释与即时编译（JIT）相结合的方式
- Java 对象通常分配在堆空间。
- Java 具有自动垃圾回收机制。
- 真题里如果把 Java 描述成“静态优化编译后直接以本地机器码为主运行”，通常是干扰项。

## 本轮补充的 Python 类型与运算知识点

### 1. Python 四类高频容器

- `list`：
  - 有序
  - 可重复
  - 可变
  - 类比 Java 的 `ArrayList`
- `tuple`：
  - 有序
  - 可重复
  - 不可变
  - 可理解为“不可改的小型记录”
- `dict`：
  - 键值映射
  - 按键访问
  - 键不能重复
  - 类比 Java 的 `Map`
- `set`：
  - 无序
  - 元素不重复
  - 类比 Java 的 `Set`

### 2. Python 常考类型判断

- 题目若问“不可变、有序、元素可重复”，优先想到 `tuple`。
- 题目若问“键值对、按键访问”，优先想到 `dict`。
- 题目若问“可变、有序、元素可重复”，优先想到 `list`。
- 题目若问“去重、集合、不重复”，优先想到 `set`。

### 3. Python 常考运算与容器操作

- 对 `list` 来说：
  - `list + list` 表示拼接
  - `list * n` 表示重复，不是逐元素数值乘法
- 示例：
  - `[1,2,3] + [4,5]` 的结果是 `[1,2,3,4,5]`
  - `[1,2] * 2` 的结果是 `[1,2,1,2]`
- 对 `tuple` 来说也有相同规律：
  - `tuple + tuple` 表示拼接
  - `tuple * n` 表示重复
- 示例：
  - `(1,2) + (3,4)` 的结果是 `(1,2,3,4)`
  - `(1,2) * 2` 的结果是 `(1,2,1,2)`
- 需要特别区分：
  - `list * list` 不合法
  - `tuple * tuple` 不合法
  - 只有“序列乘整数”才表示重复
- 所以：
  - `list × list` 不是考试里合法的数值运算
  - `tuple × tuple` 也不是

### 4. Python 字典常考操作

- `dict['name']`：
  - 表示按键取值
- `dict['age']=15`：
  - 表示按键修改值
- `del dict['age']`：
  - 表示删除某个键值对
- `del dict`：
  - 表示删除整个变量
- 真题高频误区：
  - 把 `dict` 当成 `list` 用下标访问
  - 误写成 `dict[1]`，但字典中根本没有键 `1`

### 5. Python 易混操作

- `append(x)`：
  - 把 `x` 当成一个整体元素追加到列表末尾
- `extend(x)`：
  - 把 `x` 中的元素逐个追加到列表末尾
- 示例：
  - `[1,2].append([3,4])` 结果接近 `[1,2,[3,4]]`
  - `[1,2].extend([3,4])` 结果接近 `[1,2,3,4]`
- `type(x)`：
  - 用于获取对象类型
  - 真题里可能直接考“判断对象类型的方法”

### 6. Python 中的向量与矩阵

- Python 内建类型里，没有像某些数学软件那样专门命名为“向量类型”“矩阵类型”的基础类型。
- 在不借助第三方库时：
  - 一维数据通常用 `list` 或 `tuple` 表示
  - 二维数据通常用“列表嵌套列表”表示
- 例如：
  - 向量可写成 `[1,2,3]`
  - 矩阵可写成 `[[1,2],[3,4]]`
- 但要特别注意：
  - 这些写法只是“用列表存数据”
  - 不是数学意义上自带线性代数运算规则的专门类型
- 所以在 Python 内建语义下：
  - `[1,2,3] + [4,5,6]` 是拼接，不是向量加法
  - `[1,2] * 2` 是重复，不是数乘意义下的线性代数运算
  - `[[1,2],[3,4]] * 2` 也是重复整个列表，不是矩阵数乘
- 如果题目进入真正的向量、矩阵计算语境，工程上常见做法是使用第三方科学计算库：
  - 例如 `NumPy`
- 在 `NumPy` 这类库里，数组对象更接近考试里说的“向量/矩阵”：
  - 支持逐元素加减乘除
  - 支持矩阵形状和维度相关操作
  - 支持线性代数运算
- 对软件设计师第二章来说，当前更该记住的是：
  - Python 内建 `list/tuple` 的 `+` 和 `*`，默认不是线性代数运算
  - 若不特别说明第三方库，就不要把列表当成真正的数学向量/矩阵类型

## 本轮真题暴露出的易错点

### 1. 容器类型特征混淆

- `list` 与 `tuple` 的“可变/不可变”容易混
- `dict` 与 `list` 的“按键访问/按下标访问”容易混

### 2. 运算语义混淆

- 容易把 `list + list` 误当成逐元素相加
- 容易把 `list * 2` 误当成数值乘法
- 容易误以为 `list * list`、`tuple * tuple` 合法

### 3. 编译流程阶段混淆

- 容易把：
  - 词法分析
  - 语法分析
  - 语义分析
  混成一个大概念
- 目前最需要固定下来的结论：
  - 词法分析输出记号流
  - 语法分析输入记号流
  - 类型检查属于语义分析

### 4. 语言实现特征混淆

- 容易把“解释执行”误判成“效率更高”
- 容易把 Java 的运行机制误判成传统静态本地编译模式
