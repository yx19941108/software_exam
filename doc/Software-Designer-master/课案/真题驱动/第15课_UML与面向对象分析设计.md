# 第15课：UML / 面向对象分析设计专项

## 本课定位

本课进入下午案例分析的 UML / 面向对象分析设计专项。目标不是背 UML 图形符号，而是把题干需求、用例图、类图之间的信息互相校验，稳定拿到类名、属性、关系、设计意图和扩展设计类问题的分数。

本专项主要服务于下午案例题中常见的“说明 + UML 图 + 填空 / 简答”题型。第一阶段重点放在用例图与类图组合题：先从题干名词、角色、业务动作中抽取候选类和用例，再结合图中继承、关联、聚合、组合等关系确定空缺项。

## 本课目标

1. 能从题干中识别参与者、用例、核心业务对象和对象层次。
2. 能根据类图中的泛化、关联、聚合、组合关系反推类名。
3. 能把题干中的业务描述转成类的关键属性，而不是把所有文字机械抄入答案。
4. 能区分类图中的“类名空”“属性空”“关系空”“方法空”各自的答题依据。
5. 能面对新增需求时，判断需要引入的面向对象设计思想或设计模式，并用简短文字解释原因。

## 前置知识

UML 是统一建模语言，用图来表达软件系统中的需求、结构和行为。软件设计师下午题最常考的不是完整画图规范，而是读懂已有图并补全空缺。

用例图用于描述“谁使用系统完成什么目标”。其中，参与者通常是系统外部的人、组织或外部系统；用例通常是用户能感知到的业务功能，常以动宾短语表示，例如“查询资源”“下载资源”“维护信息”。

类图用于描述系统中的静态结构。类通常来自题干中的稳定名词，例如用户、资源、订单、账户、论文等。属性通常来自对这些名词的描述，例如名称、编号、时间、状态、次数等。方法通常来自对象承担的行为，但考试中更常让考生补类名、属性或关系。

泛化关系表示“是一个”的继承层次。例如某类对象分为若干子类型时，父类承载公共属性，子类承载特有属性。关联关系表示对象之间存在业务联系。聚合和组合都表示整体与部分，但组合关系更强，部分对象通常依赖整体存在。

### UML 核心关系与图标速查

读 UML 图时，第一步不是看箭头，而是先判断“这是什么图”。同样像箭头的符号，在用例图、类图、顺序图里的含义可能不同。软考下午题中，最常见的是用例图、类图、顺序图、活动图和状态图。本课先把用例图与类图串起来，同时补充容易混淆的顺序图箭头。

#### 1. 用例图：参与者、用例、include、extend、泛化

用例图回答的问题是：外部参与者为了达到业务目标，使用系统做什么。

| 符号 / 关键字 | 中文含义 | 怎么理解 | 答题关键词 |
| --- | --- | --- | --- |
| Actor / 小人图标 | 参与者 | 系统外部与系统交互的人、组织或外部系统 | 谁使用系统 |
| 椭圆 | 用例 | 参与者希望系统完成的一个业务目标 | 做什么功能 |
| 直线 | 关联 | 参与者参与某个用例 | 谁使用哪个功能 |
| `<<include>>` | 包含关系 | 一个用例必然复用另一个公共用例 | 必做、公共步骤、复用 |
| `<<extend>>` | 扩展关系 | 在特定条件下，额外场景补充基本用例 | 可选、条件触发、补充场景 |
| 空心三角箭头 | 泛化关系 | 参与者或用例之间的父子关系 | 是一种、特殊化 |

`include` 的核心是“抽公共步骤”。如果多个用例都要执行同一个动作，就把这个动作抽成公共用例。例如创建地址簿和修改地址簿都要保存，那么保存地址簿就是被包含的公共行为。

`extend` 的核心是“条件下才发生的补充场景”。它不是流程顺序。比如某个基本用例在一般情况下可以完成，但在某种条件下需要额外处理，就把额外处理抽成扩展用例。答题时要写“在特定条件下对基本用例进行扩展或补充”，不要写成“先执行 A 再执行 B”。

用例图里的空心三角箭头通常表示泛化。箭头指向父用例或父参与者。例如“管理员”和“普通用户”都属于“用户”，箭头应指向“用户”。记忆方式是：三角箭头指向更抽象、更一般的那个概念。

#### 2. 类图：类、关联、聚合、组合、泛化、实现、依赖

类图回答的问题是：系统中有哪些类，它们保存哪些属性、提供哪些方法，类与类之间是什么结构关系。

| 符号 / 图标 | 关系名 | 中文含义 | 强弱关系 | 典型题干词 |
| --- | --- | --- | --- | --- |
| 普通实线 | 关联 Association | 两个类之间有业务联系 | 普通联系 | 使用、拥有、属于、记录、管理 |
| 实线 + 空心菱形 | 聚合 Aggregation | 弱整体-部分关系 | 部分可离开整体独立存在 | 包含、由……组成，但部分可共享 |
| 实线 + 实心黑菱形 | 组合 Composition | 强整体-部分关系 | 部分强依赖整体，整体消亡部分通常也消亡 | 由……组成，生命周期依赖整体 |
| 实线 + 空心三角箭头 | 泛化 Generalization | 继承关系 | 子类是父类的一种 | 是一种、分为、包括若干类型 |
| 虚线 + 空心三角箭头 | 实现 Realization | 类实现接口 | 实现接口或规格 | 实现、接口、规范 |
| 虚线箭头 | 依赖 Dependency | 临时使用关系 | 最弱关系 | 调用、使用、依赖某服务 |

空心菱形和黑心菱形都表示“整体-部分”，区别在强弱。

空心菱形是聚合。它表示弱拥有，部分对象可以脱离整体继续存在。例如“班级”和“学生”常被理解为聚合关系：学生属于班级，但学生离开这个班级仍然存在。

黑心菱形是组合。它表示强拥有，部分对象强依赖整体存在。例如“订单”和“订单明细”常被理解为组合关系：订单明细通常依附于订单存在，订单被删除时明细也随之消失。考试中如果题干强调“由若干部分组成”“部分生命周期依赖整体”“整体销毁部分也销毁”，优先考虑组合。

空心三角箭头在类图中表示泛化，也就是继承。箭头指向父类。判断口诀是：子类 `is a` 父类。例如学生是用户、教师是用户，那么学生和教师指向用户。

虚线加空心三角箭头表示实现，常用于“类实现接口”。如果图中出现接口名、`interface`、抽象规格，且某个类完成这个接口定义的操作，就考虑实现关系。

虚线普通箭头表示依赖。它只说明一个类在某个操作中临时使用另一个类，不表示长期拥有，也不表示继承。

#### 3. 顺序图：实心三角箭头不是类图继承

顺序图回答的问题是：对象之间按时间顺序如何发送消息。

顺序图里常见的实心三角箭头，通常表示同步消息。同步消息的意思是：发送方调用接收方的操作后，需要等待对方处理完成再继续。它表示对象间调用，不表示继承、组合或用例扩展。

所以看到“黑心三角”或“实心箭头”时，必须先看它在哪种图里：

- 在顺序图中，实心三角箭头多半是同步调用消息。
- 在类图中，继承通常是空心三角箭头，不是实心三角。
- 在用例图中，`include` 和 `extend` 依靠带关键字的虚线箭头判断，不靠三角形本身判断。

顺序图还常见返回消息，通常用虚线箭头表示。返回消息不是必须每次都画，题目更常考“谁调用谁、调用顺序、消息名是否对应题干动作”。

#### 4. 符号判断优先级

遇到 UML 图标不确定时，按这个顺序判断：

1. 先判断图类型：用例图、类图、顺序图、活动图、状态图。
2. 再看线型：实线、虚线、是否带关键字。
3. 再看端点：空心三角、空心菱形、黑心菱形、普通箭头。
4. 最后回到题干验证：是否是“是一种”、是否是“整体-部分”、是否是“必做公共步骤”、是否是“条件扩展”。

不要只看箭头方向就判断流程。UML 里的箭头更多是在表达依赖方向、继承方向、消息方向或扩展方向，未必是业务执行顺序。

#### 5. 本课常用记忆句

- `include`：多个用例都要做的公共步骤。
- `extend`：特定条件下才发生的补充场景。
- 空心三角：指向更一般的父概念。
- 空心菱形：弱整体-部分，部分可独立存在。
- 黑心菱形：强整体-部分，部分依赖整体存在。
- 虚线箭头：通常比实线关系弱，常见为依赖、实现、包含或扩展，必须看关键字和箭头端点。
- 顺序图实心箭头：消息调用，不是继承关系。

## 核心考法识别

第一类考法是类名映射。题干会给出一组业务对象，类图中用 C1、C2 等占位。解题时先找继承结构，再找题干中的分类描述。若一个概念被明确分成几种类型，通常对应父类和多个子类。

第二类考法是属性抽取。属性不是所有需求句子的堆砌，而是某个类需要长期保存的数据项。判断标准是：这个信息是否属于该类本身，是否会被系统记录，是否能帮助区分对象或支持业务处理。

第三类考法是关系判断。看到“一类对象包含若干另一类对象”“某对象来自某来源”“用户对某对象执行操作”等表述，要回到类图中寻找关联、聚合或组合位置。不要只看单个类名，要结合连线方向、数量关系和上下文。

第四类考法是新增需求下的设计扩展。题目可能要求根据新需求选择一种设计模式或面向对象机制。答题时要先抓住新需求中的变化点和对象协作关系，再用一句话说明该模式为什么匹配。

## 固定答题步骤

第一步，标出题干中的三类词：参与者、业务对象、业务动作。参与者优先对应用例图外部角色，业务对象优先对应类图中的类，业务动作优先对应用例或方法。

第二步，先解继承层次。类图中如果出现一个父类连接多个子类，先回到题干中找“包括……、分为……、有……类型”等分类句。父类通常是抽象概念或共同概念，子类通常是具体类型。

第三步，再解核心关联。对每条关联线，回到题干中找“属于、包含、来自、记录、使用、维护、下载、检索”等关系词，确认类与类之间的业务联系。

第四步，抽取属性时按“公共属性先归父类，特有属性归子类”的规则处理。若多个子类共有的信息，不要重复写到每个子类；若只有某一类对象具备的信息，应放到该子类。

第五步，处理简答题时按“需求特征 + 设计机制 + 匹配原因”的格式作答。文字要短，避免泛泛解释定义而不联系题干。

## 判分点模板

类名题通常按空给分。答题模板为：

- `C编号：类名`
- 若多个同层子类顺序可互换，应保持类名清晰，不要写成含混中文短语。
- 若题干同时给出英文名和中文名，优先使用题干中的英文类名或规范中文名。

属性题通常按类或属性组给分。答题模板为：

- `类名或C编号：属性1、属性2、属性3`
- 公共属性写在父类位置。
- 特有属性写在对应子类位置。
- 不写明显属于操作行为的动词短语。

关系或模式简答题通常按“名称 + 原因”给分。答题模板为：

- `设计机制 / 设计模式：`
- `原因：该需求体现了……，该机制能够……，因此适合用于……。`

如果题目限制字数，应先写结论，再写原因；不要先铺定义。

### 已讲评例题：地址簿管理系统中的 include 与 extend

题源：2022年上半年软件设计师考试应用技术真题，试题三。

本题给出一个地址簿管理系统。系统用于管理公司员工地址记录，地址记录包括姓名、住址、城市、省份、邮政编码、联系电话等信息。管理员可以添加、修改、删除地址记录，可以按员工姓氏字典顺序或邮政编码排序，也可以打印地址记录。系统还支持创建地址簿、打开已有地址簿、修改地址簿并保存。题目给出用例图和类图，要求补全用例、抽取类属性与方法，并解释 `extend` 和 `include`。

#### 问题1：U1～U6 用例名

参考作答：

- U1：按姓氏字典顺序排序
- U2：按邮政编码排序
- U3：创建地址簿
- U4：修改地址簿
- U5：打开地址簿
- U6：保存地址簿

其中 U1 和 U2 可以互换，因为二者都是“排序”的具体方式，图中如果位于同一组分支位置，通常不要求固定左右顺序。

解析：

先从题干中的动宾短语找用例。`排序` 不是一个单一动作，题干明确给出两种排序方式：按员工姓氏字典顺序排序、按邮政编码排序，所以 U1 和 U2 对应这两个具体用例。`创建地址簿`、`打开地址簿`、`修改地址簿`、`保存地址簿` 都是围绕地址簿本身的操作，要结合用例图位置判断。容易错的是 U3 和 U4：不要只按题干出现顺序填，要看图中它们与 `保存地址簿`、`打开地址簿` 的关系。参考答案中 U3 是创建地址簿，U4 是修改地址簿。

#### 问题2：AddressBook 与 PersonAddress 的属性和方法

参考作答：

- PersonAddress 的主要属性：姓名、住址、城市、省份、邮政编码、联系电话。
- AddressBook 的主要属性：部门名或部门编号，以及地址簿中维护的地址记录相关信息。
- AddressBook 的主要方法：添加、修改、删除、创建、打开、打印、排序等。

解析：

`PersonAddress` 是员工的一条地址记录，题干直接说“员工的地址记录包括……”，所以姓名、住址、城市、省份、邮政编码、联系电话应归到 `PersonAddress`。

`AddressBook` 是地址簿，是按部门建立并保存的一组地址记录。若从建模角度写“部门、PersonAddress 对象集合”，能表达结构含义；但考试参考答案更偏向题干原文中的可记录属性和操作，因此建议写成“部门名/编号”并补出它管理地址记录所需的方法。

方法抽取时要区分两层动作：一层是对地址记录的操作，如添加、修改、删除；另一层是对地址簿本身的操作，如创建、打开、打印、排序。只写地址记录的增删改，会漏掉地址簿对象本身的职责。

#### 问题3：extend 和 include 代入本题解释

参考作答：

- `extend`：表示扩展关系。某个基本用例在特定条件下会出现额外场景，此时由扩展用例补充基本用例。代入本题，创建地址簿和打开地址簿属于不同进入路径：新建地址簿时走创建；已有地址簿需要后续修改时，需要打开已有地址簿。这里强调的是“条件下发生的扩展场景”，不是流程图里的先后顺序。
- `include`：表示包含关系。多个用例中存在共同的、必然会复用的行为时，把公共行为抽成被包含用例。代入本题，创建地址簿和修改地址簿都需要保存地址簿，所以可以把“保存地址簿”抽出来作为公共用例，由创建和修改包含它。

详解：

`include` 比较好判断，口诀是“必做的公共步骤”。只要创建地址簿或修改地址簿最终都要保存，保存就是公共行为。此时 `保存地址簿` 不是可有可无的补充动作，而是多个用例共同依赖的步骤，所以用 `include`。

`extend` 的难点是不要把它看成流程箭头。流程上，修改已有地址簿前通常要先打开地址簿；但 UML 用例图里的 `extend` 不是用来表达“先打开，再修改”这种顺序的。它表达的是：基本用例在某些条件下会被一个额外用例补充。

代入本题，可以这样理解：如果用户处理的是新地址簿，走“创建地址簿”；如果用户处理的是已有地址簿，并准备继续修改，就需要“打开地址簿”这个额外场景。也就是说，打开地址簿是“已有地址簿需要后续处理”这一条件下出现的扩展场景。答题时可以写“在特定条件下对基本用例进行补充”，不能写“表示顺序”。

本题失分警戒：

- 把 `extend` 写成“先后顺序”会扣分。
- 只说“打开后才能修改”还不够，必须补一句“这是特定条件下发生的扩展场景”。
- `include` 要写出“公共行为 / 必然复用”，并代入“创建和修改都要保存”。

## 常见失分点

1. 把用例当成类。用例是用户目标，类是系统中长期存在或需要管理的对象。
2. 只按题干出现顺序填类名，不看类图继承和关联结构。
3. 把公共属性重复写到多个子类，导致父类属性漏答。
4. 把业务动作写成属性，例如“下载”“检索”本身通常不是属性。
5. 新增需求题只背设计模式定义，没有说明它和题干需求的对应关系。
6. 遇到英文术语时直接跳过。下午题中的英文括号通常是类名、对象名或模式名的重要提示。
7. 忽略数量关系。一个对象“包含多个”或“属于一个”往往影响类图关系判断。

## 过关标准

本课第一阶段过关标准为：面对一套用例图 + 类图组合题，能够在 25 分钟内完成全部小问，并满足以下要求：

1. 类名映射题至少拿到 70% 的空。
2. 属性抽取题能正确区分公共属性与特有属性。
3. 简答题能先写明确结论，再用题干需求解释原因。
4. 讲评后能把错因归类为“类名识别错误、属性归属错误、关系判断错误、设计机制判断错误”中的一种，而不是只记答案。
