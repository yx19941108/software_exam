# 第5章 软件工程基础知识

## 0. 本课生成口径

- 本课按真题驱动方法生成，只使用本地题库中的上午客观题。
- 这章相关题很多，为了保证可教、可复用，我采用了一个 `30` 题的代表性本地客观题池来做权值统计。
- 权值定义：`该知识点对应题数 / 30`。
- 排序规则：权值高的排前，权值低的排后。

## 一、开发过程模型与敏捷方法（权值：6/30，约20.0%）

### 2.1 这个知识点到底在讲什么

这一块研究的是：面对不同规模、不同风险、不同需求稳定度的软件项目，应该选什么开发过程模型。

软件工程里常见过程模型不是背名称，而是要知道：

- 需求稳定还是频繁变化
- 项目风险高还是低
- 系统规模大还是小
- 是否需要尽快看到可运行原型

### 2.2 核心概念详细讲解

`瀑布模型`：

- 线性推进
- 每阶段产物清晰
- 适合需求相对稳定的项目
- 缺点是变更成本高

`原型模型`：

- 先快速做一个可见原型
- 用来澄清需求
- 适合需求不清、用户一开始说不明白的场景
- 但不适合超大规模、强工程约束系统直接依赖它完成全过程

`增量模型`：

- 把系统拆成多个可逐步交付的增量
- 先交付高优先级部分
- 优点是早交付、早反馈
- 难点是“模块切分是否合理”

`螺旋模型`：

- 风险驱动
- 每一轮都要做风险分析
- 特别适合技术风险和业务风险都高的项目

`喷泉模型`：

- 面向对象开发常用
- 强调迭代、重叠和无缝回退

`XP / Scrum / Crystal` 等敏捷方法：

- `XP` 强调测试先行、简单设计、持续集成、结对编程
- `Scrum` 强调短迭代、增量交付、角色协作
- `Crystal` 强调不同项目需要不同方法论，不追求“一套流程打天下”

所以过程模型题的稳解法是：

1. 先看需求是否稳定。
2. 再看风险是否高。
3. 再看项目规模和交付节奏要求。
4. 最后匹配模型特征。

### 2.3 易错点

1. 把“需求不清”统一答成原型模型，忽略了高风险项目更偏向螺旋模型。
2. 把“增量模型”理解成“天然容易模块划分”。实际上，模块划分难恰恰是它的难点。
3. 把 `Crystal`、`XP`、`Scrum` 混为一谈。

### 2.4 真题示例

题目：

某公司计划开发一种产品，技术含量很高，与客户相关的风险也很多，则最适于采用（D）开发过程模型。

- A. 瀑布
- B. 原型
- C. 增量
- D. 螺旋

正确答案：`D`

详细解析：

- 题干中的关键信号词是：`技术含量很高`、`风险很多`。
- 一旦看到“风险高”，就要优先想到 `螺旋模型`。
- A 错：瀑布适合需求比较清楚、过程稳定的项目。
- B 错：原型更强调澄清需求，不以风险分析为核心。
- C 错：增量强调分批交付，不是风险驱动。
- D 对：螺旋模型的标志就是每一轮都围绕风险分析展开。

## 二、软件测试与评审（权值：5/30，约16.7%）

### 3.1 这个知识点到底在讲什么

这部分研究的是：怎样发现软件缺陷，以及怎样保证在缺陷暴露前就尽量把问题拦住。

它通常会考：

- 软件测试的目的
- 测试用例设计原则
- 白盒 / 黑盒方法
- 正式技术评审

### 3.2 核心概念详细讲解

软件测试的核心目标不是“证明软件没有错”，而是：

- 尽可能早、尽可能多地发现缺陷

黑盒测试：

- 不关心程序内部实现
- 关心输入、输出、需求
- 常见方法：等价类、边界值、因果图

白盒测试：

- 关心程序内部逻辑
- 常见覆盖：语句覆盖、判定覆盖、条件覆盖、路径覆盖

正式技术评审：

- 是评审，不是改代码
- 目标是尽早发现错误
- 它不用于给程序员“打绩效”

### 3.3 易错点

1. 把“测试目的”写成“证明软件正确”。这是最经典错误项。
2. 把正式评审理解成“允许高级技术人员修改程序”。
3. 把合理输入考虑了，却忘了不合理输入也要纳入测试。

### 3.4 真题示例

题目：

以下关于软件测试的叙述中，不正确的是（B）。

- A. 在设计测试用例时应考虑输入数据和预期输出结果
- B. 软件测试的目的是证明软件的正确性
- C. 在设计测试用例时，应该包括合理的输入条件
- D. 在设计测试用例时，应该包括不合理的输入条件

正确答案：`B`

详细解析：

- A 对：测试用例必须有输入，也必须有预期结果，否则无法判断对错。
- B 错：测试不是证明程序绝对正确，而是尽可能发现缺陷。
- C 对：合理输入当然要测。
- D 对：不合理输入更要测，因为鲁棒性往往就卡在这些场景。

所以这道题真正卡的是一句软件工程基本观念：

- `测试是为了发现缺陷，不是为了证明正确。`

## 三、维护与质量属性（权值：5/30，约16.7%）

### 4.1 这个知识点到底在讲什么

软件上线以后不会停止变化，所以维护和质量属性是软件工程里非常高频的基础。

这部分通常考：

- 四类维护
- 可维护性
- 软件质量子特性
- 软件工具分类

### 4.2 核心概念详细讲解

软件维护通常分四类：

1. `改正性维护`
   - 修 bug
2. `适应性维护`
   - 为适应外部环境变化而修改
3. `改善性维护`
   - 为增强性能、功能和用户体验而修改
4. `预防性维护`
   - 为后续维护做准备，提高可维护性和可靠性

判断口诀可以记成：

- 修错：改正性
- 适应环境：适应性
- 变快变强：改善性
- 提前防病：预防性

可维护性不是单点能力，而是综合质量属性。通常会和：

- 可理解性
- 可测试性
- 可修改性
- 文档质量

一起被考。

### 4.3 易错点

1. 看到“性能提升”就误判成改正性维护，实际上通常是改善性维护。
2. 误以为可维护性不受文档影响。
3. 把“适应外部环境变化”误归到改善性维护。

### 4.4 真题示例

题目：

系统交付用户使用了一段时间后发现，系统的某个功能响应非常慢。修改了某模块的一个算法使其运行速度得到了提升，则该行为属于（C）维护。

- A. 改正性
- B. 适应性
- C. 改善性
- D. 预防性

正确答案：`C`

详细解析：

- 题干没有说“程序本来错了”，而是说“响应很慢，要让它更快”。
- 这说明目标是改进性能。
- A 错：改正性是修已知错误。
- B 错：适应性是适应新环境、新平台、新接口。
- C 对：提升性能属于改善性维护。
- D 错：预防性强调提前增强可维护性，不是直接提速。

## 四、项目管理与风险、配置管理（权值：5/30，约16.7%）

### 5.1 这个知识点到底在讲什么

这部分关心的是：

- 项目怎么估计、计划、组织、控制
- 风险怎么量化
- 版本和变更怎么管

### 5.2 核心概念详细讲解

风险优先级的常见依据不是只看“概率”或只看“损失”，而是看：

- 风险暴露度 `Risk Exposure`

常见理解：

- `风险暴露度 = 风险发生概率 × 风险影响`

配置管理主要管：

- 版本控制
- 变更控制
- 配置状态记录
- 配置审计

它不直接等于“质量控制”。

### 5.3 易错点

1. 风险优先级只看概率，不看损失。
2. 把配置管理误写成质量控制。
3. 误以为加人一定能解决进度落后问题。

### 5.4 真题示例

题目：

风险的优先级通常是根据（C）设定。

- A. 风险影响（Risk Impact）
- B. 风险概率（Risk Probability）
- C. 风险暴露（Risk Exposure）
- D. 风险控制（Risk Control）

正确答案：`C`

详细解析：

- A 不完整：只看影响不够。
- B 不完整：只看概率也不够。
- C 对：风险暴露度综合考虑了“发生可能性”和“损失后果”。
- D 错：风险控制是应对措施，不是优先级计算依据。

## 五、模块设计、耦合与内聚（权值：5/30，约16.7%）

### 6.1 这个知识点到底在讲什么

这部分关心的是“模块怎么拆才算好”。

好设计通常追求：

- 高内聚
- 低耦合
- 模块规模适中
- 扇入扇出合理

### 6.2 核心概念详细讲解

`内聚`：

- 模块内部各成分之间联系有多紧密
- 越紧密越好

`耦合`：

- 模块之间依赖有多强
- 越弱越好

公共耦合：

- 多个模块共享同一个全局数据环境

通信内聚：

- 模块内部多个处理都围绕同一数据集展开

做题时要先看：

1. 模块间通过什么传信息
2. 模块内部的动作是不是围绕同一数据集

### 6.3 易错点

1. 误以为“模块越小越好”。错，规模适中才是原则。
2. 误以为公共耦合和数据耦合一样轻。错，公共耦合更差。
3. 看到“同一数据结构被读写”不会联想到通信内聚。

### 6.4 真题示例

题目：

如下图所示，模块A和模块B都访问相同的全局变量和数据结构，则这两个模块之间的耦合类型为（A）耦合。

- A. 公共
- B. 控制
- C. 标记
- D. 数据

正确答案：`A`

详细解析：

- 题干关键词是：`都访问相同的全局变量和数据结构`。
- 一旦出现“共享公共数据环境”，就对应 `公共耦合`。
- B 错：控制耦合强调传递控制信息。
- C 错：标记耦合强调通过参数表传递记录结构。
- D 错：数据耦合强调通过简单参数交换数据。

## 六、需求分析与系统设计阶段产物（权值：4/30，约13.3%）

### 7.1 这个知识点到底在讲什么

软件工程题里经常用“某阶段输出什么、不输出什么”来考阶段职责边界。

一般要区分：

- 需求分析阶段
- 概要设计阶段
- 详细设计阶段

需求分析阶段常见产物：

- 数据流图
- 数据字典
- E-R 图
- 需求规格说明

概要设计阶段常见任务：

- 体系结构设计
- 子系统划分
- 模块层次划分

详细设计阶段常见任务：

- 模块内部算法
- 数据结构细化

### 7.2 真题示例

题目：

软件开发过程中，需求分析阶段的输出不包括（D）。

- A. 数据流图
- B. 实体联系图
- C. 数据字典
- D. 软件体系结构图

正确答案：`D`

详细解析：

- A、B、C 都属于需求分析阶段典型产物。
- D 错在阶段：软件体系结构图主要属于概要设计阶段，而不是需求分析阶段。
