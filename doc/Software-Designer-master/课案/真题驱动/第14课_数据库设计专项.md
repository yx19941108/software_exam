# 第14课 数据库设计专项

## 本课定位

数据库设计是下午案例题中的固定高频题型，核心不是背数据库术语，而是把需求文字转换成可落地的数据模型。考试通常给出业务说明、部分 E-R 图或部分关系模式，要求补充实体、联系、联系类型、属性、主键、外键，并在新增需求下修改原有设计。

本课只解决下午数据库设计题的作答方法。训练题负责提供闭卷题面，课案只提供通用方法、答题模板和判分意识，不直接暴露任何配套训练题的具体答案。

## 本课目标

完成本课后，应能做到：

1. 从题干中识别实体、实体属性、联系和联系属性。
2. 判断 1:1、1:n、m:n 以及三元联系的多重度。
3. 把 E-R 图转换成关系模式，并写出主键、外键。
4. 在已有关系模式不完整时，判断缺失属性来自实体本身，还是来自联系转换。
5. 面对新增需求时，能判断是补实体、补联系、补属性，还是新增关系模式。
6. 按下午案例题的格式作答，避免多写、漏写、错写导致失分。

## 前置知识

实体是系统中需要长期记录的业务对象，通常是名词，例如用户、部门、商品、车辆、订单、门店等。实体一般有自己的标识属性，也就是能够唯一确定一个对象的属性。这个标识属性在关系模式中通常会成为主键。

属性是描述实体或联系的信息。实体属性依附于实体，例如员工的姓名、电话、岗位。联系属性依附于实体之间发生的业务关系，例如借阅日期、维修费用、处理结果、签订日期等。判断属性属于实体还是联系，关键看它是否只有在多个实体发生某种业务关系时才有意义。

联系是实体之间的业务关系，例如客户拥有车辆、部门管理职员、供应商供应零件。联系不只是连线，还要判断多重度。常见多重度有：

1. 1:1：一方一个对象最多对应另一方一个对象。
2. 1:n：一方一个对象可以对应另一方多个对象，另一方每个对象只对应这一方一个对象。
3. m:n：两边都可以多个对应多个。
4. 三元联系：一次业务事实同时依赖三个实体，不能简单拆成三个二元联系，否则会丢失业务语义。

关系模式是数据库表的文字表示，常写作 `关系名（属性1，属性2，属性3）`。主键用于唯一标识一条记录，外键用于引用其他关系模式的主键。下午题常要求标注主键、外键完整性约束。

## 核心考法识别

第一类考法是补 E-R 图。题干会给出几个实体和一张不完整图，要求根据需求描述补联系、联系类型和必要属性。解题时先找动词或业务事实，例如“管理”“采购”“销售”“安排”“阅读”“配送”，再判断它连接哪些实体。

第二类考法是补关系模式。题干给出部分关系模式和空缺项，空缺项可能来自三种来源：实体自身遗漏的属性、1:n 联系中归并到 n 端的外键、多对多或三元联系转换后需要纳入的实体主键和联系属性。

第三类考法是主键和外键。主键通常来自题干中的“唯一标识”，或由多个参与实体的主键组合而成。外键通常是从其他实体或关系模式迁入的标识属性。多写、少写、把普通属性误写成外键，都会扣分。

第四类考法是规范化判断。题目可能问某个关系模式是否存在规范性问题。常见切入点是重复属性、多值属性、部分函数依赖和传递函数依赖。答题时要说明问题现象，并给出拆分后的关系模式。

第五类考法是新增需求改造。新增需求常要求增加实体、联系、联系属性，或者把新的联系转换成新关系模式。作答时要保持原有设计不被破坏，只补必要内容。

## 固定答题步骤

第一步，圈实体。把题干中需要独立记录、具有稳定属性、可能被其他对象引用的名词列出来。不要把一次动作、一次交易过程、状态描述直接当实体，除非题干要求长期记录它并且它有独立编号或独立属性。

第二步，圈属性。把描述实体本身的信息归入实体属性；把只有在实体发生关系时才出现的信息归入联系属性。遇到日期、数量、金额、状态、结果等属性时，要特别判断它属于实体还是联系。

关系属性的判断口径是：这个信息离开本次业务关系后是否仍然有独立意义。如果仍然能描述某个对象本身，通常是实体属性；如果只有在两个或多个对象发生一次业务关系时才产生，才画在菱形联系上。例如日期、数量、结果、金额、工时这类字段常是联系属性，但实体编号、实体名称通常只是关系模式转换时进入表的外键，不要在 E-R 图中重复画成联系属性。

第三步，判联系。按题干原话判断一端能对应几个另一端。看到“一个……多个……”“每个……唯一……”“可以由多个……共同……”这类句式，要直接转成 1:n 或 m:n。三元联系要看一次业务事实是否必须同时说明三个实体才完整。

作图时可以用简化表达，只要实体、联系、连线、多重度和必要联系属性清楚即可。推荐写成 `[实体] -- 1 -- <联系> -- n -- [实体]` 这类结构；图上表达得不够清楚时，可用文字补充说明。真实考试中，题目要求“补充图”或“在图中增加”时，应优先在图形区域完成关键结构，文字说明只作为补充。

第四步，转关系模式。实体一般各自成为一个关系模式。1:n 联系通常把 1 端主键加入 n 端关系模式作为外键。m:n 联系通常单独转换成一个关系模式，包含两端主键和联系属性。三元联系通常单独转换成一个关系模式，包含三个参与实体的主键和联系属性。

第五步，写主键。实体关系的主键一般来自唯一标识。联系关系的主键通常由参与实体主键组合，必要时再加上能够区分多次发生的属性。不能机械地把所有属性都写进主键。

主键不是“所有外键集合”，而是“最小且足够唯一标识一条业务事实的字段集合”。如果题干出现“每天最多一次”“每次”“同一时段”等限定，要优先判断日期、时间、批次、序号是否才是区分多次发生记录的关键字段。外键可以很多，但不一定全部进入主键。

第六步，写外键。凡是引用其他实体主键的属性，通常就是外键。外键只说明引用关系，不等同于主键。一个属性可以既是组合主键的一部分，又是外键。

第七步，处理新增需求。新增实体先写实体属性和主键；新增联系再写联系类型和联系属性；最后把新增实体和联系转换成关系模式，并标注主键、外键。

## 判分点模板

补 E-R 图类问题按以下要点检查：

- 是否补全正确的联系名称或可接受的联系含义。
- 是否连接了正确的实体。
- 是否写出正确的 1:1、1:n、m:n 或三元联系多重度。
- 是否把联系属性放在联系上，而不是误放到某个实体上。
- 是否按题目要求“不增加新的实体”或“需要增加实体”执行。

关系模式类问题按以下要点检查：

- 关系名是否与业务语义一致。
- 属性是否完整且不过度扩展。
- 多对多联系是否单独成表。
- 三元联系是否保留为一个整体关系。
- 组合主键是否能唯一标识一次业务事实。
- 外键是否准确引用参与实体或相关关系模式的主键。

规范化类问题按以下要点检查：

- 是否指出冗余、插入异常、删除异常、修改异常中的关键问题。
- 是否能说明问题来自多值属性、部分依赖或传递依赖。
- 是否给出合理拆分后的关系模式。
- 拆分后是否保留原业务信息，不能因为拆分丢失联系。

新增需求类问题按以下要点检查：

- 是否识别新增实体及其必要属性。
- 是否识别新增联系及其联系属性。
- 是否保持原有实体和关系模式不被无故改写。
- 是否给出新增关系模式。
- 是否标注新增关系模式的主键和外键。

## 常见失分点

把联系属性误写成实体属性是高频失分点。例如数量、日期、次数、结果这类字段，往往不是某个实体天然拥有的属性，而是某几个实体发生一次业务关系时才产生的信息。

把关系模式字段全部画成 E-R 联系属性也容易失分。参与联系的实体主键会在关系模式转换时作为外键出现，但在 E-R 图阶段已经由实体连线表达，不需要再挂到菱形上。菱形旁只画题干明确属于本次业务关系的信息。

题干没有给关系额外信息时，不要强行给菱形补属性。例如“分公司拥有专卖店”只表达归属关系，没有“拥有日期、授权编号”等信息，就不必画关系属性。无依据新增属性可能按多写或错写扣分。

把 m:n 联系硬塞进某一端实体也是高频失分点。多对多联系通常不能通过在某个实体表中加一个外键解决，因为一条记录无法表达多个对应对象。正确思路通常是建立独立联系关系。

把三元联系拆成多个二元联系也容易失分。如果一次业务事实必须同时说明三个实体，拆成二元联系会表达不出“这三个对象在同一次业务中共同发生”的语义。

主键写得过多或过少都会扣分。主键过少时不能唯一标识记录，主键过多时把普通描述属性也纳入唯一标识，会破坏关系模式含义。

把所有外键都写进主键是常见误区。应先问“这几个字段是否已经能唯一确定一条业务记录”，再决定主键范围。例如同一个人每天最多发生一次的记录，通常可以用“人员标识 + 日期/时间”唯一确定，其他参与对象可能只作为外键保留。

外键漏写通常来自没有意识到某个属性是从其他实体迁入的标识。外键多写则常见于把普通名称、电话、地址等描述属性误认为引用约束。

新增需求题容易把“新增实体”和“新增联系”混在一起。判断标准是：如果题干要求长期记录一个对象及其属性，它更可能是实体；如果题干描述的是对象之间发生的业务事实，它更可能是联系。

## 过关标准

本课过关标准不是能背出概念，而是能在 20 分钟内完成一整道数据库设计案例题，并满足以下要求：

1. 能列出主要实体、联系和联系属性。
2. 能准确判断至少 80% 的联系多重度。
3. 能把 m:n 或三元联系转换成独立关系模式。
4. 能写出核心关系模式的主键和外键。
5. 面对新增需求时，能分清新增实体、联系、属性和关系模式。
6. 作答文字简洁，不大量写解释性废话，不暴露不确定的多余字段。
7. 作图题能用图形表达实体、联系、连线、多重度和必要联系属性，文字说明只补充图上不便表达的关系模式、主键和外键。
8. 能解释“为什么某个字段是联系属性，为什么某个外键不一定进入主键”。

如果一题中 E-R 图补全、关系模式补全、主外键标注三部分有两部分以上失分，说明还需要继续做同类型数据库设计训练题；如果主要失分集中在主键、外键，则下一轮应专项练“关系模式转换与完整性约束”。
