# 第6章 结构化开发方法

- 本课只基于本地上午客观题生成。
- 采用 `18` 题代表性题池做权值统计。
- 权值定义：`该知识点题数 / 18`。

## 1. 本地题池与知识点权值

1. 结构化分析基础与 DFD：`6/18`
2. 数据字典与加工规格说明：`5/18`
3. 结构化设计与模块结构：`4/18`
4. 原型、WebApp 与用户界面：`3/18`

## 一、结构化分析基础与 DFD（权值：6/18，约33.3%）

### 2.1 这个知识点到底在讲什么

结构化分析强调用一套清晰、可分层的方法，把“系统怎么处理数据”描述出来。

它的核心思想是：

- 自顶向下
- 逐步分解
- 先整体后局部

这部分高频考的是 `DFD`，也就是数据流图。

### 2.2 核心概念详细讲解

数据流图不关心界面长什么样，也不关心代码怎么写。它只关心：

- 数据从哪里来
- 到哪里去
- 在系统中经过哪些加工

DFD 的四个核心元素是：

1. 外部实体
2. 数据流
3. 加工
4. 数据存储

其中：

- `外部实体` 是系统外部的人、组织或外部系统
- `加工` 是系统内部对数据做变换的地方
- `数据流` 是数据的流动
- `数据存储` 是系统内部长期保存数据的地方

### 2.3 易错点

1. 把外部系统误判成加工。
2. 把 DFD 当成“控制流图”，其实它描述的是数据流，不是控制流。
3. 忘记 DFD 的顶层图一般只有一个总加工框。
4. 忘记父图和子图之间必须保持“平衡”，也就是子图边界上的输入输出数据流要和父图对应加工一致。
5. 把“分层越多”理解成“越抽象”。实际上 DFD 层次越往下，细节越多，抽象程度越低。

### 2.4 真题示例

题目：

结构化分析方法的基本思想是（B）。

- A. 自底向上逐步分解
- B. 自顶向下逐步分解
- C. 自底向上逐步抽象
- D. 自顶向下逐步抽象

正确答案：`B`

详细解析：

- 结构化分析的核心口号就是：`自顶向下、逐步分解`。
- A、C 把方向写反了。
- D 虽然有“自顶向下”，但强调的是抽象，不是这套方法的标准表述。

补强例题：

题目：

绘制分层数据流图（DFD）时需要注意的问题中，不包括（B）。

- A. 给图中的每个数据流、加工、数据存储和外部实体命名
- B. 图中要表示出控制流
- C. 一个加工不适合有过多的数据流
- D. 分解尽可能均匀

正确答案：`B`

详细解析：

- DFD 画的是`数据怎么流动`，不是程序里的`控制怎么跳转`。
- 所以 A、C、D 都属于画 DFD 时要注意的事，只有 B 错。
- 这道题最容易错在把“系统有控制逻辑”误听成“DFD 要把控制流也画出来”。考试不是问系统有没有控制，而是问 `DFD 这种图` 画不画控制流，答案是不画。
- 以后看到 `DFD / 数据流图 / 分层数据流图`，第一反应先锁死一句话：`只画数据，不画控制`。

## 二、数据字典与加工规格说明（权值：5/18，约27.8%）

### 3.1 这个知识点到底在讲什么

DFD 只是画框和箭头，不够细，所以必须配套两样东西：

1. 数据字典
2. 加工规格说明

### 3.2 核心概念详细讲解

`数据字典` 是对 DFD 中数据相关元素的说明书。

它通常记录：

- 数据项
- 数据流
- 数据存储
- 基本加工

它不等于：

- 外部实体总表
- 代码实现细节

`加工规格说明` 用来说明一个基本加工怎么把输入变成输出。

常用表示方法包括：

- 结构化语言
- 决策表
- 决策树

如果一个加工要根据多个条件组合决定动作，最常见答案就是：

- `判定表` 或 `决策表`

### 3.3 真题示例

题目：

数据字典是结构化分析的一个重要输出。数据字典的条目不包括（A）。

- A. 外部实体
- B. 数据流
- C. 数据项
- D. 基本加工

正确答案：`A`

详细解析：

- 数据字典核心是解释“系统内部数据相关元素”。
- 数据流、数据项、基本加工都属于常见条目。
- 外部实体通常在 DFD 中标识，但不是数据字典的核心条目类型。

## 三、结构化设计与模块结构（权值：4/18，约22.2%）

### 4.1 这个知识点到底在讲什么

结构化设计是在分析完成后，把系统落实成模块结构。

它常考：

- 结构图由什么构成
- 模块设计原则
- 模块独立性

### 4.2 核心概念详细讲解

结构图主要描述的标准成分要记精确，不要记模糊：

1. 模块
2. 调用
3. 数据
4. 控制信息
5. 转接符号

这里最容易丢分的地方是把 `控制信息` 记成 `控制`。

- `控制信息` 指模块之间传递的“状态、标志、判定结果”这一类信息。
- 它不是泛泛的“控制”这个词。
- 所以题目如果问“结构图的基本成分不包括什么”，出现 `控制` 这个选项时要特别警惕，考试常用这个措辞差一字来卡人。

结构化设计追求：

- 高内聚
- 低耦合
- 模块规模适中
- 控制域和作用域关系合理

常见耦合类型速查表：

| 耦合类型 | 描述 |
| --- | --- |
| 非直接耦合 | 两个模块之间没有直接关系，它们之间的联系完全是通过主模块的控制和调用来实现的 |
| 数据耦合 | 一组模块借助参数表传递简单数据 |
| 标记耦合 | 一组模块通过参数传递记录信息（数据结构） |
| 控制耦合 | 模块之间传递的信息中包含用于控制模块内部逻辑的信息 |
| 外部耦合 | 一组模块都访问同一全局简单变量，而且不是通过参数表传递该全局变量的信息 |
| 公共耦合 | 多个模块都访问同一个公共数据环境 |
| 内容耦合 | 一个模块直接访问另一个模块的内部数据；一个模块不通过正常入口转到另一个模块的内部；两个模块有一部分程序代码重叠；一个模块有多个入口 |
 
常见内聚类型速查表：

| 内聚类型 | 描述 |
| --- | --- |
| 功能内聚 | 完成一个单一功能，各个部分协同工作，缺一不可 |
| 顺序内聚 | 处理元素相关，而且必须顺序执行 |
| 通信内聚 | 所有处理元素都在一个数据结构的区域上 |
| 过程内聚 | 处理元素相关，而且必须按特定的次序执行 |
| 瞬时内聚（时间内聚） | 所包含的任务必须在同一时间间隔内执行 |
| 逻辑内聚 | 完成逻辑上相关的一组任务 |
| 偶然内聚（巧合内聚） | 完成一组没有关系或松散关系的任务 |

其中，`控制范围` 和 `作用范围` 要分清：

- `控制范围`：某模块本身以及它直接或间接控制到的所有下属模块。
- `作用范围`：某个判定真正影响到的那些模块集合。

判题规则只有一句：

- `作用范围应该在控制范围之内`

如果发现作用范围跑到控制范围外面了，常见改法只有两类：

1. 上移判定
2. 下移受影响模块

再补一条这轮也涉及到的高频边界：

- `概要设计` 重点看整体模块划分、模块间接口、总体结构质量。
- `详细设计` 重点看模块内部算法、模块内数据结构、数据库物理设计、输入输出与界面等实现级细节。

所以：

- “模块之间的接口设计”更偏`概要设计`
- “算法、模块内数据结构、数据库物理设计”更偏`详细设计`

### 4.3 真题示例

题目：

结构化设计方法中使用结构图来描述构成软件系统的模块以及这些模块之间的调用关系。结构图的基本成分不包括（D）。

- A. 模块
- B. 调用
- C. 数据
- D. 控制

正确答案：`D`

详细解析：

- 结构图里强调的是模块、调用、数据、控制信息和转接符号。
- 这道题不是说结构图里完全没有“控制相关内容”，而是说标准成分叫`控制信息`，不叫笼统的`控制`。
- 你这轮会觉得 “每个都对”，根源就在这里：课案如果只写“控制信息”而不专门提醒它和“控制”不是同一个考试术语，就很容易在考场上把 D 也当对。
- 以后见到这种题，先把标准表述在脑子里完整背出来，再去和选项逐字对照。

补强例题：

题目：

在划分模块时，一个模块的作用范围应该在其控制范围之内。若发现其作用范围不在其控制范围内，则（D）不是适当的处理方法。

- A. 将判定所在模块合并到父模块中，使判定处于较高层次
- B. 将受判定影响的模块下移到控制范围内
- C. 将判定上移到层次较高的位置
- D. 将父模块下移，使判定处于较高层次

正确答案：`D`

详细解析：

- 先别急着记答案，先看判题逻辑：`哪里做判断，哪里就要能控制到所有被这个判断影响的模块`。
- 如果控制不到，就说明“作用范围跑出了控制范围”。
- 正确修法只有两类：
  - 让判定站得更高，也就是`上移判定`
  - 让受影响模块站得更低，也就是`下移受影响模块`
- A、B、C 本质上都在把“影响范围”和“控制范围”重新收拢到一起。
- D 错在它把“父模块下移”说成一种通用修法，但这并不是结构化设计里解决该问题的标准处理方式。

## 四、原型、WebApp 与用户界面（权值：3/18，约16.7%）

### 5.1 这个知识点到底在讲什么

结构化开发方法并不只会考 DFD 和结构图，还会带一点：

- 原型在需求澄清中的作用
- WebApp 的分析与设计
- 用户界面设计

### 5.2 核心概念详细讲解

`原型` 最适合做两类事：

- 帮助用户把需求说清楚
- 帮助设计者验证方案是否合理

它不擅长：

- 指导代码优化
- 替代完整工程设计

### 5.3 真题示例

题目：

以下关于系统原型的叙述中，不正确的是（C）。

- A. 可以帮助导出系统需求并验证需求的有效性
- B. 可以用来探索特殊的软件解决方案
- C. 可以用来指导代码优化
- D. 可以用来支持用户界面设计

正确答案：`C`

详细解析：

- A 对：原型最大价值之一就是澄清需求。
- B 对：探索型原型本来就是为了验证解决思路。
- C 错：原型不负责代码级优化。
- D 对：界面原型是最常见用途之一。
